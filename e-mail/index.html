Email prototype — summary for Spark

Context

This note summarizes the email workflow prototype I'm building so you can see the kinds of problems I'm trying to solve, and where Spark (or something like it) might eventually fit.

I was inspired by Daniel Levitin's The Organized Mind, especially the idea that our brains work better when information is grouped and externalized rather than left as a mixed, scrolling inbox. Email, as designed in the 1960s, gives us a single chronological stream; my goal is to translate that stream into structured, meaningful units of work.

What I'm trying to build

High-level goal: a system that processes inbound email for me and turns it into structured data, so that:

  • My inbox is not my to‑do list.
  • I can see tasks, financial activity, and relationship follow‑ups in calm, dedicated views.
  • I can answer "what do I need to do now?" without scrolling a chaotic inbox.

Concretely, I'm building:

  • A small web app that talks to a personal data repository and some scripts that read email from my provider.
  • The scripts parse incoming mail and write structured data into that data repository:
    • tasks — things I need to do.
    • transaction-inbox — parsed financial transactions.
    • career/natework — people, notes, and follow‑ups.
  • The web UI gives purpose‑specific views: Tasks, Natework (contacts/follow‑ups), and Finances.

In other words, email is just one input channel into a structured, version‑controlled life‑management system, not the place where I manage work.

How it differs from today's email clients

Most modern clients (including Spark) do helpful things like splitting out Notifications / Newsletters / New senders, and providing Ai features primarily focused on outbound composition.

Where I still feel a gap:

  • I don't primarily need help writing email; I need help processing it into the way I live:
    • "This message should become a task in my task system."
    • "This message is a financial transaction that should join my transaction log."
    • "This exchange represents a relationship I want to nurture and follow up on."
  • Even helpful groupings still leave me with the mental work of deciding what each message means, turning that into a concrete unit (task, transaction, contact), and making sure it doesn't get lost.

The prototype uses scripts to parse patterns, translate mail into structured data (tasks, transactions, contact updates), and let a repository handle history and sync. I'm not giving an Ai agent free access to my live inbox; I'm using Ai to build tools I call on demand, with outputs stored as data I control.

What I'd love from a client like Spark

  • First‑class "translate into data" actions: "Make this a task," "Recognize this as a bill and send to finance log," "Update this person and their next follow‑up."
  • Opinionated inbox views based on meaning: "Things you need to do," "Money that moved," "People you might want to reconnect with."
  • Extensibility: Let advanced users define patterns and outputs (e.g. webhooks, append to a file); let external tools plug in as first‑class destinations.

In short: I'm building an email‑driven personal operating system where email is the raw material, and structured data + calm UIs are how I actually manage life. If Spark moves in this direction, I'd be happy to walk through the prototype and workflows with your product team.
